# 1. 프로젝트 개요

## 1.1 문제 정의

데이터 기반 프로젝트를 시작할 때 공통적으로 다음 문제가 발생한다.

- 데이터셋과 Open API가 여러 사이트에 흩어져 있다.
- 검색 결과만으로는 품질과 활용 가능성을 판단하기 어렵다.
- 프로젝트 목적에 맞는 데이터 조합을 찾는 데 시간이 오래 걸린다.
- API 응답 구조, 인증 방식, 호출 제한을 직접 확인해야 한다.

현재 서비스는 개별 데이터 자원 검색은 제공하지만, **프로젝트 목적에 맞는 적합도 판단과 데이터셋/API 조합 추천**까지는 충분히 제공하지 않는다.

## 1.2 해결 목표

SODA는 사용자가 프로젝트 목적을 자연어로 입력하면 다음을 제공하는 시스템을 목표로 한다.

- 관련 **데이터셋 + Open API** 후보 추천
- 각 후보의 구조와 활용 난이도 요약
- 프로젝트 유형별 적합도 점수
- 실제 적용을 돕는 가이드

## 1.3 이번 리팩토링의 방향

멘토 피드백을 반영해 인프라와 데이터 처리 구조를 단순화한다.

| 항목 | 이전 방향 | 현재 방향 |
| --- | --- | --- |
| 대용량 처리 | HDFS + Spark 중심 | MVP에서는 제외 |
| 비동기 처리 | 큐 필요성만 논의 | `Celery + RabbitMQ`로 명확화 |
| 캐시/상태 | Redis 범용 사용 | 캐시, rate limit, lock 용도로 한정 |
| 벡터 저장 | 별도 벡터 DB 검토 | `PostgreSQL + pgvector` 우선 |
| 서버 역할 | 준-HA와 분산을 동시에 시도 | `S1 = 서비스`, `S2 = 비동기/운영` |

# 2. 핵심 기능

## 2.1 프롬프트 기반 검색

사용자 입력 예시:

- "서버 로그 이상 탐지 모델 만들고 싶어"
- "날씨 기반 배달 수요 예측 앱을 만들고 싶어"
- "실시간 환율 정보가 필요한 핀테크 서비스를 만들고 싶어"

처리 흐름:

```text
사용자 프롬프트
  -> 질의 임베딩 생성
  -> pgvector 검색
  -> LLM 기반 추천/분석 생성
  -> 적합도 분석 결과 반환
```

핵심 원칙:

- 검색은 실제로 수집한 메타데이터를 기반으로 수행한다.
- LLM은 검색 결과를 바탕으로만 분석을 생성한다.
- 할루시네이션을 줄이기 위해 RAG 구조를 유지한다.

## 2.2 데이터 자원 후보 추천

### 데이터셋 추천

- 의미 기반 검색
- 태스크 기반 필터링
- 상위 N개 후보 추출

### Open API 추천

- 카테고리 기반 필터링
- 인증 방식, 응답 형식, 호출 제한 비교
- 서비스 활용 가능성 판단

## 2.3 구조 자동 분석

### 데이터셋 분석

- 샘플 수, 컬럼 수 요약
- 데이터 타입 판별
- 라벨 구조 확인
- 결측치 및 기본 통계 확인

### Open API 분석

- 응답 스키마 파싱
- 필드 수와 타입 요약
- 인증 방식, 호출 제한 수집
- 응답 속도와 가용성 기본 측정

## 2.4 프로젝트 적합도 점수

프로젝트 유형을 다음처럼 나눠 평가한다.

| 유형 | 예시 | 주요 평가 기준 |
| --- | --- | --- |
| AI 모델 학습 | 이미지 분류, 감성 분석 | 라벨 명확성, 데이터 규모, 클래스 균형 |
| 서비스 개발 | 날씨 앱, 금융 서비스 | API 안정성, 응답 속도, 호출 제한 |
| 데이터 분석 | 트렌드 분석, 시계열 리포트 | 데이터 완결성, 시계열 연속성, 결측치율 |
| 하이브리드 | 실시간 추천/분석 서비스 | 데이터 품질 + API 가용성 |

## 2.5 프로젝트 가이드 제공

- 전처리 또는 연동 시 체크할 점
- 데이터셋 + API 조합 제안
- 대안 후보 비교
- 활용 시나리오 설명

# 3. 시스템 아키텍처

## 3.1 전체 구조

```text
[외부 데이터 소스]
  -> [Celery Beat]
  -> [RabbitMQ]
  -> [Celery Worker / Crawler]
  -> [PostgreSQL + pgvector] / [S3]
  -> [FastAPI RAG]
  -> [웹 UI]
```

## 3.2 선택한 설계 원칙

### 원칙 1. 작게 시작한다

- HDFS, Spark 같은 대형 분산 처리 구성은 MVP에서 제외한다.
- 먼저 완성 가능한 구조를 만든 뒤 필요하면 확장한다.

### 원칙 2. 책임을 나눈다

- `S1`은 `prod active + dev active`를 함께 담당한다.
- `S2`는 `async / ops`를 담당한다.
- 현재 구조는 `S1` 단일 ingress를 사용한다.

### 원칙 3. 브로커와 캐시를 분리한다

- `RabbitMQ`: 작업 브로커
- `Redis`: 캐시, rate limiting, lock, 짧은 상태값

### 원칙 4. 상태 저장은 단순하게 유지한다

- 메타데이터와 벡터 검색은 `PostgreSQL + pgvector`
- 원본 파일은 `S3 / MinIO`

## 3.3 서버 역할

### S1

- 단일 `Nginx`
- `prod Frontend / Spring Boot API / FastAPI RAG API`
- `dev Frontend / Spring Boot API / FastAPI RAG API`
- `dev PostgreSQL / Redis / MinIO`

### S2

- RabbitMQ
- Celery Beat
- Celery Worker
- crawler
- Jenkins
- Prometheus / Grafana / Loki

## 3.4 비동기 수집-임베딩 파이프라인

```text
Celery Beat
  -> collect task enqueue
  -> RabbitMQ collect_queue
  -> Celery Worker
  -> source fetch / normalize
  -> PostgreSQL 저장
  -> S3 저장
  -> embed task enqueue
  -> RabbitMQ embed_queue
  -> Celery Worker
  -> 임베딩 생성
  -> pgvector 갱신
```

이 구조의 장점:

- 수집과 임베딩을 분리할 수 있다.
- 일부 소스 실패를 독립적으로 재시도할 수 있다.
- 사용자 요청 경로와 배경 작업 경로를 분리할 수 있다.

# 4. 기술 요소별 역할

## 4.1 Celery

- 주기 작업 스케줄링
- 수집 작업 실행
- 임베딩 작업 실행
- 재색인 및 후처리

## 4.2 RabbitMQ

- 작업 전달
- 작업 대기열 관리
- 실패 작업 재처리 경로 확보
- collect, embed, reindex 큐 분리

## 4.3 Redis

- 검색 결과 캐시
- rate limiting
- 중복 실행 방지 lock
- 짧은 TTL 상태 저장

## 4.4 PostgreSQL + pgvector

- 데이터셋/Open API 메타데이터 저장
- 벡터 검색 인덱스 저장
- 서비스 도메인 데이터 저장

## 4.5 S3 / MinIO

- 원본 파일 저장
- 응답 샘플 저장
- 추후 재처리용 아카이브 보관

# 5. MVP 단계 정의

## 최소 MVP

- 데이터셋/Open API 메타데이터 수집
- `RabbitMQ + Celery` 파이프라인 구축
- `pgvector` 기반 검색
- 프롬프트 입력 -> 추천 결과 리스트 제공
- 기본 검색 UI
- 로그인 연동

## 1차 MVP

- 프로젝트 적합도 점수
- 데이터셋 상세 구조 분석
- Open API 상세 스펙 분석
- 상세 분석 화면

## 2차 MVP

- 리뷰/평점
- 채팅 히스토리
- 데이터 조합 공유 게시판
- 수집 현황 대시보드

## 3차 MVP

- 데이터 신뢰도 검증
- 비교 화면
- 시각화 고도화

# 10. 구현 경로 및 Compose 계약

## 10.1 도메인별 Compose 구조

- `frontend/`
  - `docker-compose.yml`
- `docker-compose.dev.yml`
- `docker-compose.prod.yml`
- `backend/`
  - `docker-compose.yml`
- `docker-compose.dev.yml`
- `docker-compose.prod.yml`
- `data-platform/`
  - `docker-compose.yml`
- `docker-compose.dev.yml`
- `docker-compose.prod.yml`
  - `docker-compose.worker.yml`

## 10.2 데이터 플랫폼 구성

- `data-platform/api/`: FastAPI RAG
- `data-platform/crawler/openapi/`: Open API 수집
- `data-platform/crawler/dataset/`: 데이터셋 수집
- `data-platform/docker-compose.worker.yml`: Celery worker/beat, crawler

`spark/`, `hdfs/` 디렉터리는 저장소에 남아 있어도 MVP 운영 범위에는 포함하지 않는다.

## 10.3 인프라 경계

- `infra/`는 Nginx, Jenkins, monitoring, 공통 네트워크 설정을 담당한다.
- 앱 실행 책임은 각 도메인 compose가 가진다.
- 현재 목표 상태는 **S1 서비스 + S2 비동기/운영** 구조다.
